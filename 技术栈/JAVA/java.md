


# 1. 回源工作原理
客户端发起请求：当用户访问一个网站时，客户端（例如浏览器）会向CDN节点发送一个请求，请求的URL指向CDN加速域名。

CDN节点查找缓存：CDN节点首先会检查自身的缓存中是否存在请求的资源。如果存在，CDN节点会直接返回缓存的资源给客户端，从而实现快速响应。

回源请求：如果CDN节点的缓存中不存在请求的资源，CDN节点会根据配置的回源策略，向源站发起回源请求。

回源获取资源：源站接收到CDN节点的回源请求后，根据请求的URL和参数，生成并返回相应的资源。

缓存资源：CDN节点在获取到源站返回的资源后，会将资源缓存到自身的缓存中，以便下次请求时能够直接返回缓存的资源。

返回响应：CDN节点将获取到的资源返回给客户端，完成整个请求-响应过程。

## 9.1. mysql数据库事务隔离级别
### 9.1.1. 读未提交（Read Uncommitted）：
    特性：事务可以读取其他事务尚未提交的数据。
    问题：可能会出现脏读（Dirty Read）、不可重复读（Non-Repeatable Read）、幻读（Phantom Read）。
### 9.1.2. 读已提交（Read Committed）：
    特性：事务只能读取其他事务已提交的数据。
    问题：避免了脏读，但仍可能出现不可重复读和幻读。
### 9.1.3. 可重复读（Repeatable Read）（MySQL InnoDB默认隔离级别）：
    特性：在同一个事务内的所有读操作都会看到同样的数据（即事务开始时的快照）。
    问题：避免了脏读和不可重复读，但在某些情况下仍可能出现幻读。
### 9.1.4. 串行化（Serializable）：
    特性：事务被完全隔离，相当于事务一个接一个地执行。
    问题：避免了脏读、不可重复读和幻读，但性能开销较大，通常会导致更多的锁定和阻塞。

## 9.2. MySQL MVCC的工作原理
    MySQL的多版本并发控制（MVCC）是一种并发控制方法，用于确保数据库系统在并发环境下能够高效且正确地执行事务。MVCC的核心思想是通过维护数据的多个版本，使得读操作和写操作可以并发执行，从而提高数据库系统的并发性能。

### 9.2.1. 数据版本管理：
    每行数据都有两个额外的隐藏列，用于存储版本信息：trx_id（事务ID）和roll_pointer（回滚指针）。
    trx_id标识创建或最后一次修改该行的事务ID。
    roll_pointer指向该行的上一个版本，形成一个版本链。
### 9.2.2. 读操作（快照读）：
    快照读是一种读取一致性视图的方法，不会对数据库加锁。
    每个事务在开始时会拍摄一个时间点的快照，之后的所有读操作都基于这个快照。
    读取行数据时，根据事务的快照时间点和行数据的trx_id决定读取哪个版本的数据。
    这样，读操作不会阻塞写操作。
### 9.2.3. 写操作（当前读）：
    当前读包括了锁定读操作，如SELECT ... FOR UPDATE和SELECT ... LOCK IN SHARE MODE，以及插入、更新和删除操作。
    当前读会锁定读取的行，确保写操作的并发安全性。
    写操作会生成一个新的数据版本，并更新trx_id和roll_pointer，指向前一个版本。
### 9.2.4. 事务提交和回滚：
    事务提交后，修改的数据版本变为最新的版本。
    如果事务回滚，通过roll_pointer可以找到之前的版本，恢复数据。
### 9.2.5. 垃圾回收：
    MySQL通过后台的purge线程，定期清理不再需要的旧版本数据，释放存储空间。

# 15. Redis集群主要有三种模式
    主从复制模式（Master-Slave）、哨兵模式（Sentinel）和Cluster模式。
    主从复制模式：适用于数据备份和读写分离场景，配置简单，但在主节点故障时需要手动切换。
    哨兵模式：在主从复制的基础上实现自动故障转移，提高高可用性，适用于高可用性要求较高的场景。
    Cluster模式：通过数据分片和负载均衡实现大规模数据存储和高性能，适用于大规模数据存储和高性能要求场景。

# 16. 接口幂等
    确保接口幂等性（Idempotency）是设计可靠系统的关键，尤其是在分布式系统和微服务架构中。接口幂等性意味着对同一操作的多次调用会产生相同的结果，即使在请求过程中出现重复操作。
# 17. 乐观锁
    乐观锁假设数据冲突不常发生，因此不会在操作前加锁，而是在提交数据时检查数据是否发生冲突。如果检测到冲突，则会重试或返回错误。乐观锁通常通过版本号机制实现。
    乐观锁适用于冲突少、读多写少的场景，优势在于并发性能高，但可能会导致重试。
# 17. 悲观锁
    悲观锁假设数据冲突经常发生，因此在读取数据时就会加锁，直到事务结束。这种方式确保了数据的一致性，但可能会降低并发性能。
    悲观锁适用于冲突多、写多读少的场景，优势在于数据一致性高，但可能会导致阻塞和性能下降。
